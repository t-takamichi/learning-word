# STEP 2: タスクを順に実装

**目的**: 計画書のタスク（T1→Tn）を、依存順に1つずつ実装する。

## 実施サイクル

各タスクについて、以下のサイクルを回す。

```
[タスク Ti]
  ① 計画書の「タスク詳細」を読む      → 対象ファイル・分類(新規/移行/置換)・トークン参照・注意点
  ② 適用ルールを再確認               → Atomic依存・トークン参照・reduced-motion
  ③ 実装する                        → コンポーネント作成 / 既存移行 / CSS のトークン置換
  ④ そのタスクを検証する             → 計画書記載の検証方法を実行
  ⑤ 通れば次のタスク Ti+1 へ          → 落ちたら原因を直してから進む
```

## 実装時の遵守事項

- **Atomic 依存方向を守る**（`component-structure.md` R-ATOM-01）
  - `Pages → Templates → Organisms → Molecules → Atoms → Tokens` の一方向
  - Atom は Molecule 以上を import しない
- **トークン参照を徹底する**（R-ATOM-02）
  - 色・角丸・影は `--berry-*` 等を参照。`*.module.css` に生HEXを書かない
  - トークン未定義なら `index.css` の `:root` に追加してから参照する
- **移行タスクは既存を活かす**
  - 新パス（`components/atoms/…` 等）へ再配置し、import 参照の更新箇所を洗い出す
  - 移行中も型チェックが通る状態を保つ（旧ファイルは切替完了まで残置可）
- **機能を壊さない**: 既存フック・API のシグネチャを変えない。演出は結線のみ
- **iOS/アクセシビリティ**: サウンドは解錠・vibrate存在チェック、アニメは reduced-motion 簡略版
- **計画外の変更をしない**: スコープを広げない（Could を作らない）。必要ならユーザーへ確認

## タスクごとの検証例

| 対象 | 検証コマンド / 方法 |
|------|--------------------|
| デザイントークン / CSS置換 | `typecheck` 通過＋ブラウザ目視（背景・面・文字がトークン色） |
| Atom / Molecule | `typecheck` 通過＋仮配置で目視（variant/状態切替） |
| サウンド（useSound） | iOS Safari 実機で初回タップ後に即時再生・vibrateエラーなし |
| 演出の結線 | Good タップで音×視覚×言葉が200ms以内に発火するか目視 |
| reduced-motion | OS設定 or DevTools で reduce にしてアニメ簡略化を確認 |

## 進捗管理
- タスク数が多い場合は TaskCreate / TaskUpdate で進捗を可視化してもよい
- 各タスク完了時に、何を実装し検証結果がどうだったかを簡潔に残す

## 完了条件
- 計画書の全タスク（T1〜Tn）が実装され、各検証が通っていること
- 全タスク完了後、STEP 3 の検証・報告へ進む
