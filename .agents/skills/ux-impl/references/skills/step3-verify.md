# STEP 3: 検証・受け入れ基準の確認・報告

**目的**: UXフェーズ全体が計画どおり動くことを確認し、正直に報告する。

## 実施内容

### 全体ビルド・型チェック
- プロジェクト全体で型チェックを実行する
  ```
  （プロジェクトの typecheck スクリプト。例: yarn typecheck / bun run typecheck = tsc --noEmit）
  ```
- 生HEX混入チェック：`*.module.css` にトークン以外の HEX が残っていないか grep で確認
  （`:root` 定義と管理画面のエラー色 `#E5484D` を除く）

### 受け入れ基準（AC）を確認
- 計画書 `phase-N.md` の「受け入れ基準（AC）」を上から確認する
- 各項目に実際にチェックを入れられるか、動作で確認する
- 対応する `docs/spec/design/ux/acceptance-criteria.md` の項目と突き合わせる

### フェーズゴールの動作確認
- `phase-N.md` の「ゴール」を実際に動かして確認する
- 目視: `/run`（アプリ起動）で対象画面を表示し、Berry化・演出・文言を確認
- iOS実機: サウンド解錠・セーフエリア・vibrate無害（サウンド/演出フェーズで必須）
- アクセシビリティ: `prefers-reduced-motion` で演出簡略化、本文コントラスト AA

### 正直な報告
実装結果をユーザーへ報告する。以下を必ず含める。

- 実装したタスク（T1〜Tn）と、それぞれの検証結果
- 受け入れ基準チェックリストの充足状況（満たした / 未達）
- 型チェック・目視の結果。**iOS実機で未検証の項目は「未検証」と明示**する
- 落ちたテスト・スキップした確認があれば、その内容を明示
- 「完了」と言えるのは検証が通ったときのみ

## 完了条件
- `rules/quality-gate.md` の全チェック項目がパスすること

## 完了後の案内
```
UX-<N> 実装完了。次のフェーズへ進む場合:
/ux-impl <次のUXフェーズ番号>
```
