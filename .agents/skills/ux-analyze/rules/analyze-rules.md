# Analyze Rules — UI/UX分析ルール (マルチエージェント版)

分析時に、各エージェント（Lead, PM, Designer, CEO）が必ず守るルール。

## R-AN-00: 顧客属性・ターゲットの合意形成
- 分析を開始する前に、必ずユーザー（オーナー）にターゲット属性や重視する体験についてインタビューを行い、合意を得ること。
- 合意が得られるまで、分析成果物の書き出しを行ってはならない。

## R-AN-01: コード上の事実に基づく（PM義務）
- 指摘する課題には、単なる印象論ではなく、必ず **「対象ファイルパス・行数・該当するCSSセレクタやロジック」** のようなコード上の具体的な事実を併記すること。

## R-AN-02: 合意したペルソナの目で見る
- 評価基準は、自分の好みや固定の前提ではなく、STEP 0 でユーザーと合意したターゲット属性（例：一般の言語学習者、共有デバイスでの家族利用など）の心理や利用環境に準拠すること。

## R-AN-03: 良い点も壊さない
- 課題だけでなく、維持すべき良い点も必ず記録すること。（改善の際に元の良さを失わないため）

## R-AN-04: 改善は"仮説"として出す
- 分析フェーズで結論を断定しない。設計フェーズで検証・具体化する前提で、改善は「仮説」として提示する。

## R-AN-05: 優先度を必ず付ける（Lead責任）
- PMの見積もる「コスト」と、デザイナー/CEOの評価する「体験価値（インパクト）」の2軸で優先度を判定し、高インパクト×低コストの「Quick Win」を明示すること。

## R-AN-06: 実装を勝手に変えない
- このスキルは分析専用。`src/**` のコードを直接編集しない（Write対象は `docs/spec/design/ux/analysis/` のみ）。

## R-AN-07: ディズニー品質の「加点（Plussing）」提案（デザイナー義務）
- 単に「不具合がない」だけではなく、「どうすればより楽しくなるか」「イージングやトランジション、フィードバックで感動を与えられるか」という加点要素の提案を必ず1つ以上含めること。

## R-AN-08: 感情曲線と継続設計（CEO義務）
- 「起動から退場まで」の感情の起伏を Mermaid グラフ等を用いてビジュアルで表現し、感情の谷（離脱点）を特定すること。
- ゲーミフィケーション（Octalysisフレームワークなど）に基づく継続性の仕組みが機能しているかを評価すること。
