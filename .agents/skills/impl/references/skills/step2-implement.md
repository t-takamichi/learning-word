# STEP 2: タスクを順に実装

**目的**: 計画書のタスク（T1→Tn）を、依存順に1つずつ実装する。

## 実施サイクル

各タスクについて、以下のサイクルを回す。

```
[タスク Ti]
  ① 計画書の「各タスク詳細」を読む   → 対象ファイル・シグネチャ・注意点
  ② 適用ルールを再確認              → docs/rules の該当規約
  ③ 実装する                       → ファイル作成 / 編集
  ④ そのタスクを検証する            → 計画書記載の検証方法を実行
  ⑤ 通れば次のタスク Ti+1 へ         → 落ちたら原因を直してから進む
```

## 実装時の遵守事項

- **レイヤー依存方向を守る**（`docs/rules/architecture.md`）
  - 共有型 → Domain → Infrastructure → Application → Presentation → Frontend
- **型安全を徹底する**: `any` を避け、Hono は `zValidator`、共有型は `shared/`
- **既存スタイルに合わせる**: 命名・フォーマット・コメント密度を周囲に揃える
- **計画外の変更をしない**: スコープを広げない。必要ならユーザーへ確認する

## タスクごとの検証例

| 対象 | 検証コマンド / 方法 |
|------|--------------------|
| 型定義 | `tsc --noEmit` が通る |
| ドメイン/ユースケース | 単体テスト（あれば） |
| API | `curl` でエンドポイントを叩き期待レスポンスを確認 |
| フロント | 画面を表示し操作を確認（`/run` や `/verify` を活用可） |

## 進捗管理
- タスク数が多い場合は TaskCreate / TaskUpdate で進捗を可視化してもよい
- 各タスク完了時に、何を実装し検証結果がどうだったかを簡潔に残す

## 完了条件
- 計画書の全タスク（T1〜Tn）が実装され、各検証が通っていること
- 全タスク完了後、STEP 3 の検証・報告へ進む
