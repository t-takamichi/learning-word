# STEP 1: 設計書の自動同期 (design-sync)

**目的**: ユーザーの自然言語の指示から影響を受ける設計書を特定し、自律的にアップデートする。

## 実施内容

1. **指示の解析**:
   - ユーザーが要求した「機能追加」「UI/UXの変更」「ロジックの修正」がどのような内容であるかを読み取る。
2. **対象設計ファイルの特定**:
   - `docs/spec/design/` (UI/UXなら `docs/spec/design/ux/`) の中から、影響を受けるマークダウンファイルを特定する。
   - 適切なファイルがない場合（新規機能の場合）は、`docs/spec/design/[feature-name].md` のように新規設計ファイルを作成する。
3. **設計書の差分更新**:
   - 特定した設計ファイル（例: `docs/spec/design/README.md` または `docs/spec/design/flashcard.md` など）に、新規要件や変更後の仕様を追記・修正する。
   - この際、全体のMoSCoW優先度やユーザーストーリー、受け入れ基準（AC）も自動で更新する。
4. **差分（変更内容）の出力と報告**:
   - どの設計ファイルをどのように書き換えたかの差分をメモリ上に保持し、次のステップ（フェーズ定義更新）のインプットとする。
   - ユーザーに対し、「設計書 `docs/spec/design/xxx.md` を更新しました」と報告する。

## 完了条件
- 影響を受ける設計書が、ユーザーの指示内容を網羅した形で正しく更新・保存されていること。
- 更新後のマークダウン構造が崩れていないこと。
