# Flow Rules — 安全ガードレール・修復ポリシー

`flow` スキルを実行中、エージェントは以下のルールを厳格に遵守しなければなりません。

---

## 1. 起動時の安全確認 (Git Clean Check)

- **ルール**: パイプライン実行を開始する前に、必ず Git のステータスを確認すること。
- **動作**: 
  - `git status --porcelain` を実行し、未コミットの変更があるかチェックする。
  - もし変更がある場合、自動的に `git stash save "flow-auto-backup-<TIMESTAMP>"` を実行して変更を安全に退避させる。
  - 退避が完了したことをユーザーに報告し、パイプラインの終了後（またはロールバック後）に `git stash pop` で元に戻す準備をしておく。

---

## 2. 設計書と計画書の同期整合性

- **ルール**: コード実装に移る前に、必ず設計書（`docs/spec/design/`）と実装計画書（`plan.md`）のアップデートを完了させること。
- **動作**:
  - 設計書の差分に基づき、影響するフェーズを正しく特定する。
  - 実装計画書（`plan.md`）を書き換える際は、既存のタスクのIDや順序を破壊せず、差分としてのタスク（例: `T-ADD-1`）を新規に挿入・追記する。
  - ファイル編集時は必ずバックアップを作成し、書き込みに失敗した場合は直ちに復元する。

---

## 3. 自己修復（セルフリペア）ループの制限

- **ルール**: コード修正後の検証（ビルドやテスト）でエラーが出た場合、自己修復は最大 **3回** までとする。
- **動作**:
  - エラーログを解析し、原因を特定してコードを再修正する。
  - 再度ビルド・テストを実行する。
  - このループを最大3回まで繰り返し、3回失敗した場合はそれ以上の自動修正を断念する。

---

## 4. 安全なロールバック

- **ルール**: 自己修復が最終的に失敗した場合、または重大な未定義エラーが発生した場合は、速やかにコードを元のクリーンな状態に差し戻すこと。
- **動作**:
  - 変更を加えた対象のコードファイルに対し、`git checkout -- <file>` または `git reset --hard` を実行し、起動前のクリーンな状態に戻す。
  - 設計書や実装計画書（`plan.md`）に関しては、変更した内容を残しておくか、または同様にロールバックするかをログに明示し、ユーザーに引き継ぐ。
  - 退避させていた Git の Stash がある場合は、`git stash pop` を実行してユーザーの元の手動変更を復元する。
  - 失敗したエラーログの詳細と、ロールバックが完了した状態をユーザーにチャットで明確に報告する。

---

## 5. 開発履歴の記録 (flow-history.md)

- **ルール**: パイプラインの実行が完了（またはロールバック）した際、必ずその開発履歴を `docs/spec/agent/flow-history.md` に追記すること。
- **動作**:
  - 新規の実行ログをファイルの末尾に追記し、Markdown の見出し（`## 実行ログ - [TIMESTAMP]`）で区切る。
  - ログには以下の内容を含める：
    - 実行日時とユーザー指示。
    - **STEP 1**: 更新された設計書ファイルパスと、仕様変更箇所の要約。
    - **STEP 2 & 3**: 影響を受ける実装フェーズ番号、および更新された計画書（`plan.md`）のパスと追加タスクID。
    - **STEP 4**: 修正されたソースコードファイルパス、および検証実行結果（ビルド/テスト合格ステータス、自己修復リトライ回数）。
  - ロールバックが発生した場合は、その旨と失敗原因となったエラーログの要約も履歴に記録する。
