# STEP 6: 受け入れ基準（Acceptance Criteria）の定義（PM）

**目的**: 各機能の「完了」を明確に定義する。実装者との認識齟齬を防ぐ。

## 実施内容

### ACフォーマット
```
機能名: [機能名]

AC1: [前提条件] のとき、[操作] をすると、[期待される結果] になること
AC2: [前提条件] のとき、[操作] をすると、[期待される結果] になること
AC3: （エラーケース）[異常条件] のとき、[操作] をすると、[エラー処理] になること
```

### 記載例（フラッシュカード機能）
```
機能名: フラッシュカード表示

AC1: セッション開始時、APIから10問の単語データが取得され、カード表面（英語）が表示されること
AC2: カード領域をタップすると、裏面（翻訳・例文）が表示されること
AC3: 「Good」ボタンをタップすると、review_countが+1され、次のカードに遷移すること
AC4: 「Again」ボタンをタップすると、incorrect_countが+1され、ステータスが`weak`になること
AC5: 10問すべて回答すると、セッション完了画面が表示されること
```

### 各機能にACを定義する
- Must have の機能は全てACを作成する（省略不可）
- ACが書けない = 要件が曖昧 → STEP 1・2 に戻る

## 出力先
各機能の詳細設計ファイル `docs/spec/design/[feature-name].md` に追記
