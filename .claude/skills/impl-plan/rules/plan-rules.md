# Plan Rules

実装計画書の作成時に必ず守るルール。違反する計画は受け付けない。

---

## ドキュメントルール

### R-PL-DOC-01: 計画書は docs/spec/impl/steps/<フェーズ番号>/plan.md に置く
- フェーズ番号ごとにディレクトリを切る（例: `docs/spec/impl/steps/1/`）
- 1フェーズ=1つの計画書（`plan.md`）を必須とする

### R-PL-DOC-02: フェーズ定義・設計へトレースできること
- 計画書の冒頭に対応する `phase-N.md` と設計書へのリンクを記載する
- フェーズのスコープ外の作業を計画に混入させない

### R-PL-DOC-03: 計画は実装者がそのまま着手できる具体度にする
- 「認証を実装」ではなく「`src/server/middleware/basicAuth.ts` を作成し〜」まで書く
- 対象ファイルパス・追加/変更の別・主要な型やシグネチャを示す

---

## タスク分割ルール

### R-PL-01: 1タスク=1つの明確な成果物にする
- 成果物（ファイル/関数/コンポーネント/マイグレーション）が1つに定まる粒度にする

### R-PL-02: タスクはレイヤー依存順に並べる
- `docs/rules/architecture.md` の依存方向に従う
- 原則: 共有型 → Domain → Infrastructure → Application → Presentation → Frontend

### R-PL-03: 各タスクに検証方法を添える
- 「どうやって動作確認するか」（コマンド・手動確認・テスト）を必ず書く
- 検証不能なタスクは分割し直す

### R-PL-04: 受け入れ基準（AC）を計画のチェックリストへ落とす
- フェーズ定義・設計書のACを、計画末尾のDoneチェックリストへ転記する

### R-PL-05: コーディング規約への準拠を明記する
- 適用される `docs/rules/` のルール名を各タスクに紐づける（例: 型安全・DIパターン）

### R-PL-06: 見積もりと順序番号を付ける
- 各タスクに連番（T1, T2, ...）を振り、実装順を一意にする
