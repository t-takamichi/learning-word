# Design Rules

設計時に必ず守るルール。違反する設計は受け付けない。

---

## ドキュメントルール

### R-DOC-01: docs/spec/design/README.md が最初に存在すること
- 全体設計の入口として `docs/spec/design/README.md` を必ず最初に作成する
- 詳細設計は README.md からリンクされること
- README.md なしに詳細設計ファイルだけ作らない

### R-DOC-02: シーケンス図は Mermaid 記法で書く
- テキストでシーケンス図を表現し、バージョン管理可能な状態にする
- 画像ファイルのみでの保存は禁止
- コードブロック: ` ```mermaid `

### R-DOC-03: 設計ドキュメントには「なぜ」を必ず書く
- 「Aを採用した。」ではなく「BではなくAを採用した理由は〜」と書く
- 将来の開発者が読んで意図を理解できること

---

## PMルール

### R-PM-01: ユーザーストーリーなき機能設計は禁止
- 全機能に対して最低1つのユーザーストーリーを書く

### R-PM-02: 受け入れ基準（AC）を設計フェーズで定義する
- 実装フェーズに入る前に各機能のACを確定させる
- ACが書けない機能は「設計未完了」として扱い実装に進まない

### R-PM-03: MoSCoW優先度を全機能に付ける
- Must have のみでMVPを定義できる状態にする
- Won't have を明示してスコープクリープを防ぐ

### R-PM-04: 成功指標（KPI）を仮定義する
- 測定可能な指標を設計段階で少なくとも1つ定める

---

## エンジニアルール

### R-ENG-01: iPhone/Safari を第一優先に設計する
- Web Speech API のiOS制限を設計に織り込む
- CSS は iPhone のセーフエリア（`env(safe-area-inset-*)`）を考慮する

### R-ENG-02: データモデルは変更コストを見積もって設計する
- SQLite のマイグレーション戦略を記載する

### R-ENG-03: APIは型安全に設計する
- Hono の `zValidator` を使ったスキーマ定義を前提とする
- フロントエンドとバックエンドで型を共有する構成を検討する
