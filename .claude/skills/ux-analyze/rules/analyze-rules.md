# Analyze Rules — UI/UX分析ルール

分析時に必ず守るルール。

## R-AN-01: 事実ベースで書く
- 「ダサい」「良くない」等の感想で終わらせない
- 必ず **事実 → なぜ問題か → 改善仮説** の3点セットで書く

## R-AN-02: 対象ユーザーの目で見る
- 評価基準は常に「20代ベトナム人女性・片手・隙間時間・iPhone Safari」
- 自分の好みではなく、ユーザーの感情・行動で判断する

## R-AN-03: 良い点も壊さない
- 課題だけでなく、維持すべき良い点も記録する（改善で失わないため）

## R-AN-04: 改善は"仮説"として出す
- 分析フェーズで結論を断定しない。設計（/ux-design）で検証する前提で仮説として提示

## R-AN-05: 優先度を必ず付ける
- インパクト×コストで優先度を付け、Quick Win を明示する
- 全部同じ重みで並べない

## R-AN-06: 実装を勝手に変えない
- このスキルは分析専用。`src/**` を編集しない（Write対象は `docs/spec/design/ux/analysis/` のみ）
