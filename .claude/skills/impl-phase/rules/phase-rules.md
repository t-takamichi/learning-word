# Phase Rules

フェーズ設計時に必ず守るルール。違反する設計は受け付けない。

---

## ドキュメントルール

### R-PH-DOC-01: docs/spec/impl/phase/README.md が最初に存在すること
- フェーズ設計の入口として `docs/spec/impl/phase/README.md` を必ず最初に作成する
- 各フェーズ詳細（`phase-N.md`）は README.md からリンクされること

### R-PH-DOC-02: 全フェーズは設計書へトレースできること
- 各フェーズには対応する設計書のリンク（`docs/spec/design/[feature].md`）を記載する
- 設計書に存在しない機能を勝手に追加しない（追加が必要なら設計へ差し戻す）

### R-PH-DOC-03: フェーズ間の依存関係を Mermaid で図示する
- `graph LR` などでフェーズの前後関係・並行可能性を可視化する

---

## フェーズ分割ルール

### R-PH-01: 各フェーズは単独で動作確認できる単位にする
- 「このフェーズ完了時点で何が動くか」を必ず定義する
- 動作確認不能な中途半端な塊を1フェーズにしない

### R-PH-02: レイヤーの依存方向を守る
- `docs/rules/architecture.md` の依存方向に反するフェーズ順序にしない
- 例: Infrastructure より先に依存先の Domain 型を用意する

### R-PH-03: 最初のフェーズは Walking Skeleton にする
- Phase 1 は全レイヤーを薄く貫く「最小で動く1機能」にする
- 環境構築・DBスキーマ・1本のAPI・1画面が通ることをゴールにする

### R-PH-04: MoSCoW優先度をフェーズ順序に反映する
- Must → Should → Could の順にフェーズを並べる
- Won't have はフェーズに含めない

### R-PH-05: 技術リスクの高い項目を早いフェーズに寄せる
- `docs/spec/design/risks.md` の「高×高」項目を早期フェーズで検証する

### R-PH-06: 1フェーズの粒度を適切に保つ
- 大きすぎる場合はサブフェーズに分割する
- 小さすぎる（数行）場合は近接フェーズに統合する
