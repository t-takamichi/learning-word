# STEP 3: 要件ごとのフェーズ分割

**目的**: 層マップと依存関係をもとに、要件を「単独で動作確認できるフェーズ」に分割する。

## 実施内容

### フェーズ分割の方針
- **Phase 1 は Walking Skeleton**: 全レイヤーを薄く貫く最小の1機能を通す
  （環境構築 + DBスキーマ + 1本のAPI + 1画面）
- 以降は **Must → Should → Could** の順に、機能単位でフェーズを積む
- リスクの高い項目は早いフェーズに寄せる
- 各フェーズは「完了時点で何が動くか」を必ず定義する

### 各フェーズ詳細ファイルの作成
フェーズごとに `docs/spec/impl/phase/phase-N.md` を作成する。以下の構成を必須とする。

```markdown
# Phase N: [フェーズ名]

## ゴール（このフェーズ完了時に動くもの）
- 例: セッションAPIが単語10件を返し、カード画面に1枚目が表示される

## 対応する設計
- [機能名](../../design/[feature].md)

## スコープ（このフェーズでやること）
| 層 | 実装項目 |
|----|---------|
| Domain | ... |
| Infrastructure | ... |
| Application | ... |
| Presentation | ... |
| Frontend | ... |

## スコープ外（このフェーズでやらないこと）
- 次フェーズ以降に回す項目を明示

## 依存フェーズ
- 前提となるフェーズ番号（なければ「なし」）

## 完了条件（Doneの定義）
- [ ] 動作確認できる観点をチェックリストで書く
- [ ] 対応する設計書のAC（受け入れ基準）を満たす

## 想定リスク
- `docs/spec/design/risks.md` から関連リスクを転記
```

## 出力先
`docs/spec/impl/phase/phase-1.md`, `phase-2.md`, ... （フェーズ数だけ作成）
