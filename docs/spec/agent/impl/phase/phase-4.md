# Phase 4: 逆同期と全体のE2E検証

## ゴール（このフェーズ完了時に動くもの）

- `/flow --reverse` コマンドにより、手動で変更されたコードの `git diff` を自動解析し、その内容を `spec/design/` 配下の該当設計書に自動で逆反映できる
- 設計変更の指示（インプット）から、最終的なコード実装・検証完了（アウトプット）まで、一切の手動介入なしで一気通貫でパイプライン（E2E）が動作する

---

## 完了条件（Doneの定義）

- [ ] 手動でコード変更（例: UIのラベル変更など）を加えた状態で `/flow --reverse` を実行した際、`spec/design/` 配下の関連設計ファイルに変更内容が記述されること
- [ ] テストシナリオ（新規機能の追加指示）をインプットとして `/flow` を起動し、手動コマンド介入なしで設計書更新、計画更新、コード実装、型チェック、テストパスまでが自動完遂すること
- [ ] 完了時に全体の進捗や完了ステータスを示す報告がチャット上に出力されること

## 想定リスク

| リスク | 優先度 | 対策 |
|--------|--------|------|
| **R4: コードの差分（diff）が大きすぎる場合、設計書に反映すべき要点が絞り込めず、設計書が不要なコード片で汚染される** | 中×中 | `reverse-parser.ts` にて、コード変更から「どのようなユーザー体験（UX）や機能の変更が行われたか」を2〜3行の抽象的な要約に圧縮してから設計書へ反映させる。コードそのものの diff を直接設計書に貼り付けるような挙動は禁止とする。 |
