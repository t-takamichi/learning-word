# Phase 1: コマンド基盤と設計自動更新

## ゴール（このフェーズ完了時に動くもの）

- `flow` スキルの定義（`SKILL.md`）が追加され、エージェントにスキルとして認識されている
- ユーザーの自然言語による「機能追加・変更」の指示を受け取り、`spec/design/` 配下の該当する設計書（例: `README.md`）を自動的に更新するステップが実行できる

---

## 対応する設計

- [オーケストレーター設計書](../../orchestrator.md) (背景・要件定義 AC1, AC2)

---

## スコープ（このフェーズでやること）

| 層 | 実装項目 |
|----|---------|
| **エントリー/インフラ** | `.agents/skills/flow/SKILL.md` (スキル起動定義) |
| **エージェント定義** | `.agents/skills/flow/references/agent/persona.md` (統合開発調整者としてのペルソナ) |
| **共通ルール** | `.agents/skills/flow/references/rules/flow-rules.md` (フローの基本ルール、自己修復ループの最大回数等のポリシー) |
| **ステップ定義** | `.agents/skills/flow/skills/step1-design-sync.md` (ユーザーの指示をもとに設計書を更新するプロンプトステップ) |
| **共通スクリプト** | `.agents/skills/flow/scripts/utils.ts` (Gitの差分取得やファイル検索などを補助する軽量 TypeScript ヘルパー) |

---

## スコープ外（このフェーズでやらないこと）

- 実装計画書（`plan.md`）の自動マージ・差分適用（Phase 2）
- 影響を受けるフェーズ（`phase-N.md`）の自動特定（Phase 2）
- ソースコードの自動修正とビルド・テスト検証（Phase 3）
- エラー検出時の自己修復ループおよび Git ロールバック（Phase 3）
- 手動コード変更から設計書への逆反映機能（Phase 4）

---

## 依存フェーズ

なし

---

## 完了条件（Doneの定義）

- [ ] `.agents/skills/flow/SKILL.md` が正常に配置されている
- [ ] ユーザーから「/flow〇〇の機能を追加して」という変更指示を受けた際、エージェントが `flow` スキルのペルソナを確立すること
- [ ] テスト用（あるいは実際）の設計書 `spec/design/README.md` に対して、指示内容を組み込んだ差分更新が自動で行われ、ファイルが正しく保存されること
- [ ] 型エラーやシンタックスエラーがないこと (`.agents/skills/flow/scripts/` 配下の TypeScript スクリプトが `yarn tsc` やコンパイルをパスすること)

---

## 想定リスク

| リスク | 優先度 | 対策 |
|--------|--------|------|
| **R1: エージェントがプロンプト定義だけで複雑な差分同期ロジックを行おうとして失敗する（安定性の欠如）** | 高×中 | 設計書の検索や更新を単なるプロンプト任せにするのではなく、更新前後の差分を確認する軽量ヘルパースクリプト（`scripts/utils.ts`）をエージェントに提供し、スクリプト実行とプロンプト推論をハイブリッドで組み合わせる。 |
