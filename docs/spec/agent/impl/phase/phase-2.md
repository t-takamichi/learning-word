# Phase 2: 計画書（plan.md）の自動同期

## ゴール（このフェーズ完了時に動くもの）

- 設計書の変更差分（Markdown）を解析し、影響を受ける実装フェーズおよび実装計画書（`plan.md`）のファイルパスを自動特定できる
- 特定された `plan.md` 内のタスクや受け入れ基準に対し、既存の構成を崩すことなく変更要件に対応するタスク（例: `T-ADD-1`）を自動マージして保存できる

---

## 対応する設計

- [オーケストレーター設計書](../../orchestrator.md) (機能要件 AC3)

---

## スコープ（このフェーズでやること）

| 層 | 実装項目 |
|----|---------|
| **ステップ定義** | `.agents/skills/flow/skills/step2-phase-sync.md` (影響分析・フェーズ定義更新ステップ) |
| **ステップ定義** | `.agents/skills/flow/skills/step3-plan-sync.md` (実装計画書 `plan.md` 自動マージ・更新ステップ) |
| **共通スクリプト** | `.agents/skills/flow/scripts/impact-analyzer.ts` (設計差分から影響フェーズをマージ・マッピングするためのロジックを提供する TypeScript スクリプト) |

---

## スコープ外（このフェーズでやらないこと）

- 実際のソースコードの自動実装、テスト・型チェック検証の自動実行（Phase 3）
- エラー検出時の自己修復ループおよび Git ロールバック（Phase 3）
- 手動コード変更から設計書への逆反映機能（Phase 4）

---

## 依存フェーズ

- Phase 1: コマンド基盤と設計自動更新

## 完了条件（Doneの定義）

- [ ] 設計書に差分が加わった際、関連する `spec/impl/steps/<フェーズ番号>/plan.md` が自動特定されること
- [ ] `plan.md` に対して、既存のタスク一覧（T1, T2...）を維持したまま、新しい変更用のタスクが適切な位置に自動挿入されること
- [ ] 計画書のマークダウン構造が破壊されることなく、有効なフォーマットのまま保存されること
- [ ] `impact-analyzer.ts` が型エラーなしでビルドおよび実行できること

## 想定リスク

| リスク | 優先度 | 対策 |
|--------|--------|------|
| **R2: 実装計画書 (plan.md) のMarkdownフォーマットが破壊される** | 中×中 | ASTレベルでの厳密なマークダウン・パーサーによる書き換えではなく、エージェント自身のMarkdown編集能力と `impact-analyzer.ts` の正規表現によるセクション検出を組み合わせたハイブリッドでマージを実行する。また、書き換え前に必ず元のファイルのバックアップ（`.bak` または一時ファイル）を取得し、書き込みに失敗した場合は直ちに復元する。 |
