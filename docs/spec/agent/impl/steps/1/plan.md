# Phase 1 実装計画書: コマンド基盤と設計自動更新

## 1. 参照

- フェーズ定義: [phase-1.md](../../phase/phase-1.md)
- 設計（オーケストレーター）: [orchestrator.md](../../orchestrator.md)

---

## 2. このフェーズのゴール

- `flow` スキルの定義（`SKILL.md`）が追加され、エージェントが呼び出し可能になること
- ユーザーの指示から `spec/design/` 配下の適切な設計書が自動特定され、自律的に差分更新ができること
- エージェント実行のための基本ペルソナ、安全ルール、および軽量ヘルパースクリプト（`utils.ts`）が用意されていること

---

## 3. 前提・依存

- 依存フェーズ: なし（最初のフェーズ）
- 事前準備:
  - ワークスペースが git リポジトリであり、`git` コマンドが実行可能であること
  - Node.js/Bun 環境にて TypeScript スクリプトがコンパイル・実行可能であること

---

## 5. 各タスク詳細

### T1: スキル定義ファイルの作成

- **ファイル**: `.agents/skills/flow/SKILL.md`（新規）
- **内容**:
  * `/flow` コマンドをエントリポイントとし、指示テキストを受け取る引数仕様を定義する。
  * `references/agent/persona.md` と `references/rules/flow-rules.md` を読み込んだ上で、`skills/step1-design-sync.md` にチェーンする基本構造を定義する。
- **検証**: スキルが正常に読み込まれること。

---

### T2: ペルソナと基本ルールの作成

- **ファイル**:
  * `.agents/skills/flow/references/agent/persona.md`（新規）
  * `.agents/skills/flow/references/rules/flow-rules.md`（新規）
- **内容**:
  * **persona.md**: 全体の設計・計画・実装・検証を繋ぐ統合コーディネーターとしてのペルソナ（PM×エンジニア×QA）。
  * **flow-rules.md**:
    * 処理開始時に必ず Git がクリーンであることを検証すること。
    * 自己修復の最大回数を3回に制限すること。
    * 失敗時は強制ロールバックを行い人間に引き継ぐこと。
- **検証**: エージェントがペルソナとルールを理解し、クリーンチェックを意識すること。

---

### T3: 設計同期ステップ定義の作成

- **ファイル**: `.agents/skills/flow/skills/step1-design-sync.md`（新規）
- **内容**:
  * ユーザーからの指示テキストをパースし、どの機能（画面・API・ロジック）に関するものかを判断する。
  * `spec/design/` (および UI/UX に関するものは `spec/design/ux/`) を検索し、該当するファイルを特定する。
  * 指示内容を反映するように、設計ファイルを自律的にアップデートする（ファイルの作成または replace_file_content の使用）。
- **検証**: テスト用マークダウンファイルに対する自律的な書き換えが完了すること。

---

### T4: 共通スクリプト (`utils.ts`) の作成

- **ファイル**: `.agents/skills/flow/scripts/utils.ts`（新規）
- **内容**:
  * `isGitClean()`: `git status --porcelain` を実行し、未コミットの変更がないかチェックする。
  * `findDesignFiles()`: `spec/design/` 内のマークダウンファイルをリストアップする。
- **検証**: スクリプトを直接実行し、正しい結果が返ってくることを確認。

---

### T5: テスト実行と初期動作確認

- **やること**:
  * `spec/design/README.md` にダミーの要件追加（例: 「テスト用項目」など）を `/flow` で指示して、T1〜T4を組み合わせた自律的な書き換えを検証する。
- **検証**: 設計書が自動で更新されることを確認する。
