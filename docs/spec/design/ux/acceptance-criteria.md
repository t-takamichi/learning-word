# UI/UX 受け入れ基準 (Acceptance Criteria)

本ドキュメントは、実装フェーズに入る前に満たすべき、UI/UXデザイン品質の具体的な受け入れ基準（AC）を定義します。開発完了後の検収はこのチェックリストに基づいて行われます。

---

## 1. ビジュアル・デザイントークン領域 (Visual & Tokens)

*   **[ ] デザイントークンの完全適用 (R-UX-03)**
    *   すべての UI コンポーネントの `*.module.css` にて、生の色コード（HEX, rgb）やマジックナンバーが直接記述されておらず、必ず `--berry-500` や `--radius-card` 等のデザイントークン変数を使用していること。
*   **[ ] 真っ黒の不使用 (R-UX-04)**
    *   `#000000` や `black` のカラー指定が CSS 内に存在しないこと。テキストの最も暗い色は `--ink-900` (`#4A2C3A`) であること。
*   **[ ] コントラスト比の確保 (R-UX-04)**
    *   白背景上のテキスト色は `--ink-900` または `--ink-700` が適用され、コントラスト比 4.5:1 (AA) 以上を満たしていること。
    *   ピンク背景（`--berry-500` 等）の上に白文字テキストが重なっていないこと（白文字を重ねる場合は背景を `--berry-600` 以上の濃いピンクとする、またはテキスト色を `--ink-900` にする）。
*   **[ ] モバイル Safari ズーム防止**
    *   単語追加フォーム、単語セット作成フォームのすべての入力要素（`input`, `textarea`, `select`）の `font-size` が **16px** 以上に設定されており、iOS Safari でフォーカス時に画面が強制ズームされないこと。
*   **[ ] フォーカスインジケータの適用 (アクセシビリティ)**
    *   すべてのインタラクティブ要素（ボタン、アバター、入力フィールド）がフォーカスされた際、ブラウザ標準の青い枠線を排除し、Berryテーマのピンク光彩を表す `--focus-ring` トークンが正しく表示されること。

## 2. モチベーション・演出領域 (Motivation & Copy)

*   **[ ] 正解時フィードバックの即時同期 (R-UX-05)**
    *   学習画面で「Good」をタップした際、Correct音＋瞬間バイブレーション＋スパークル演出が **200ms以内** の遅延で同時に開始されること。
*   **[ ] 失敗（Again）の低刺激ケア (R-UX-06)**
    *   Again タップ時に不快なブザー音が鳴らず、マイルドな低音 `again` が鳴り、画面に点滅や強い警告色の演出が入らないこと。
*   **[ ] 誤操作リカバリー (Undo) の検証**
    *   学習中、Undoボタンをタップした際にカードが裏面（未回答）に戻り、コンボ数および学習進捗率（ヘッダーインジケータ）が正しく1つ前の状態に安全に巻き戻ること。
    *   Undoボタンが、誤操作を防ぐために Good/Again などの主要な回答ボタンから十分離れた位置（カードの隅等）に配置されていること。
*   **[ ] 連続正解（コンボ）の非ブロッキング動作 (R-UX-07)**
    *   5コンボや10コンボの演出が発生した際、画面を覆って操作をロックするモーダル（2.4秒画面ロック等）が表示されず、非同期のパーティクルが飛び散るだけで、ユーザーは操作をそのまま続行できること。
*   **[ ] 応援形マイクロコピーの完全適用**
    *   単語の削除、単語セットの削除確認、単語数およびセット数上限到達時のエラー表示において、従来の命令形コピー（「削除しますか？」等）ではなく、設計仕様に定義された「おわかれする？」「いまはここまで！よくがんばったね🌸」などのやさしい応援形マイクロコピーが適用されていること。
*   **[ ] アバター選択簡単ログインの動作と既存パスワード互換性**
    *   共有デバイス環境において、登録済みアバター（🍓や🍎など）が一覧で表示されること。アバターをタップ後、テンキーまたは「キーボードで入力する」切り替えにより、数字・英数字どちらのパスワードを持つユーザーも支障なくログインできること。また、従来通りおなまえ入力による直接ログインへのフォールバックが完備されていること。
*   **[ ] セッション完了後の継続ループ設計**
    *   リザルト画面に「本日の目標単語数」に対する達成ゲージが表示され、目標達成時に「目標クリア！🌸」の追加お祝いが発動すること。
*   **[ ] 柔軟な訳語入力（ベトナム語/日本語のどちらか片方での登録受容）**
    *   単語登録・編集時に、英単語（english）が入力されていれば、訳語（ベトナム語 / 日本語）は両方必須ではなく**どちらか片方の入力のみでも正常に登録・更新できること**（両方空の場合のみエラーとすること）。
    *   学習画面や単語リストにおいて、片方の訳語が空の場合でも画面崩れや例外クラッシュを起こさずに正しく表示・切り替えができること。

## 3. サウンド・ハプティクス領域 (Sound & Haptics)

*   **[ ] 音声自動再生ブロックの確実な解錠 (スタートボタン) (R-UX-08)**
    *   学習画面 `StudyPage.tsx` の表示直後、画面全体を覆う「学習をスタート！🍓」ボタンが表示されること。これをタップした瞬間に `AudioContext.resume()` が呼び出され、アンロックのクリック音が鳴ってからカード学習が開始されること。これによって、自動再生の初回発音が iOS で無音にならないこと。
*   **[ ] 音なし（サイレント）での体験成立**
    *   端末がマナーモード等で効果音が聞こえない状態でも、視覚的な演出（トースト、スパークル、マスコットの表情変化）のみで、進捗や回答の判定が完全に理解できること。
*   **[ ] ミュート状態の引き継ぎと保存**
    *   ミュート切り替えボタン（🔊 / 🔇）をタップすると、セッション中の効果音が即座に消音/再生されること。また、その状態が `localStorage` に保存され、リロード後や次回起動時にも引き継がれること。
*   **[ ] カード遷移・操作時の先行音声即時キャンセル**
    *   例文や単語のTTS音声再生中に「できた！」「もう一度」の選択やUndo、スワイプ操作、および次のカードへの遷移が発生した際、**直前の音声が即座にキャンセル（停止）され、新旧の音声が混ざらないこと**。
*   **[ ] バイブレーションの安全性フォールバック**
    *   `navigator.vibrate` を呼び出す前に必ず存在チェックが行われており、iOS Safari 等の非対応ブラウザでエラーにならず、静かに処理をスルーできること。

## 4. コンポーネント構造領域 (Atomic Design)

*   **[ ] 依存方向の一貫性 (R-ATOM-01)**
    *   コンポーネントの依存関係が一方向（Pages → Templates → Organisms → Molecules → Atoms）になっており、下位層が上位層を import していないこと。
*   **[ ] 状態とビジネスロジックの閉塞 (R-ATOM-03)**
    *   Atoms および Molecules は、ドメイン状態や API 通信、React Query などのフックを持たない「Pure Component（純粋な見た目部品）」であること。
    *   API アクセスやデータ管理は、Organisms または Pages 階層のみで行い、Props を通じて下位へ流していること。

---

## 5. アクセシビリティ・制約領域 (Mobile Constraints)

*   **[ ] タップターゲットの確保**
    *   すべてのクリック/タップ可能な要素（ボタン、リスト内の削除/編集ボタン、タブ）のタップターゲット領域が、最低でも **44px × 44px** 以上確保されていること。
*   **[ ] `prefers-reduced-motion` の遵守 (R-UX-07)**
    *   OS の「視覚効果を減らす」設定が有効な場合、フリップアニメーションの速度が高速化され、激しいキラキラ演出などの画面エフェクトがフェードイン/アウトなどの静的な切り替えに置き換わること。
