# リスク評価

## リスクマトリクス

| # | リスク | 影響度 | 発生確率 | 対策 |
|---|--------|--------|----------|------|
| R1 | iOSでHTML5 Audio (TTS) が自動再生できない | 高 | 高 | ユーザーアクション起点でのみアンロック可能。最初の画面タップ時に共有 `HTMLAudioElement` の `play()` を同期呼び出ししてアンロックし、使い回す |
| R2 | 非同期処理やuseEffectからの `audio.play()` がブロックされる | 高 | 高 | ユーザー操作起点でアンロック済みの共有 `HTMLAudioElement` の `src` を書き換えて再生する |
| R3 | iPhoneのサイレントスイッチで音声が無音になる | 中 | 高 | UIに「音が出ない場合はサイレントモードをOFFに」の説明を常時表示 |
| R4 | SQLiteの同時書き込み競合 | 低 | 低 | WALモード有効化（`PRAGMA journal_mode=WAL`） |
| R5 | スコープクリープ（機能追加要求） | 中 | 高 | Won't have を設計書で明示し、追加要求は次バージョンに積む |
| R6 | `100vh` がiOSのアドレスバーで意図どおりにならない | 中 | 高 | `100dvh` を使用。フォールバックで `height: -webkit-fill-available` |
| R7 | Web Speech API でベトナム語音声が利用不可 | 低 | 中 | 今回は英語音声のみ実装。ベトナム語音声はV2検討 |
| R8 | Basic認証の資格情報が平文で送信される | 中 | 低 | HTTPS必須。本番環境ではTLSを終端するリバースプロキシ（nginx等）を前提とする |
| R9 | Bunのネイティブ SQLite APIが将来変更される | 低 | 低 | 抽象レイヤー（db.ts）でラップし、変更影響を1ファイルに閉じ込める |
| R10 | ユーザー削除APIが無認証で実行可能かつ全ユーザー名が露出する | 高 | 高 | GET一覧APIを廃止し、「名前＋PIN/合言葉」でログインする形に変更。削除は本人のみに制限し、削除時に合言葉（PIN）の入力による本人確認を必須とする。 |

## リスクレベルの定義

| レベル | 定義 |
|--------|------|
| **高** | 対処しないとアプリが動かない or ユーザー体験が著しく損なわれる |
| **中** | 対処しないと品質・セキュリティ・スケジュールに影響する |
| **低** | 許容範囲内だがモニタリング継続 |

## 高×高リスクの事前検証（技術スパイク）

### R1 / R2: iOS HTML5 Audio (TTS) の動作検証

**検証内容**:
1. iPhoneのSafariでユーザーアクション起点の同期処理で `HTMLAudioElement` のアンロックを実行し動作するか確認
2. 自動再生や非同期処理からの `audio.play()` がアンロック後に動作するか確認
3. サイレントスイッチON時の挙動確認

**検証タイミング**: 実装フェーズのPhase 1（コア実装）の最初に実施  
**合格基準**: iPhoneのSafariで音声再生が3回以上連続して正常に動作すること

### R3: サイレントスイッチ対策の検証

**検証内容**: iPhoneのサイレントスイッチON時に `HTMLAudioElement` が無音になるか確認  
**対策**: UIに説明メッセージを追加。追加の対策（AudioContext経由）はV2検討

## 技術的負債の許容判断

| 判断 | 内容 |
|------|------|
| 許容する | Vanilla JSではなくReactを使用することによる初期バンドルサイズ増加（~100KB）|
| 許容する | Basic認証の単純さ（シングルユーザー想定のため） |
| 許容しない | SQLiteのWALモード設定漏れ（書き込み競合の原因になりうる） |
| 許容しない | 環境変数なしのハードコード認証情報を本番環境で使用すること |
